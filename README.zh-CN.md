# P-Person Travel Assistant

一个为“不想提前做攻略的人”设计的 AI 即时旅行助手。

这个项目不是传统意义上的行程规划工具，而是一个面向真实出行场景的即时决策产品：用户到了某个地方之后，只需要告诉系统“我现在在哪”“我现在想做什么”，系统就能立刻给出附近可去的地方，并把推荐结果和地图联动展示出来。

如果把它放在作品集里，我希望它传递的不是“我做了一个地图 App”，而是：

- 我能从用户类型出发定义产品
- 我能把模糊自然语言转成可执行的搜索逻辑
- 我能把 LLM、地图服务、前端交互和工程稳定性串成一个完整可运行的系统

## 一、项目背景

大部分旅游产品都默认用户是“J 人”：

- 出发前会做大量攻略
- 会提前查店、查景点、查路线
- 会比较不同方案，再决定去哪

但真实旅行中，有另一类非常高频的用户：

- 到了再想
- 走到哪算哪
- 不想切换多个 App 来回找信息
- 想要的是“此时此刻附近有什么值得去的”

这个项目就是针对这类用户设计的。

我把它定义成一个给 **P 人** 用的旅行助手，核心不是“规划”，而是“即时推荐”。

## 二、产品定位

### 产品一句话

一个根据用户**当前位置**和**即时意图**，实时推荐附近去处的 AI 旅行助手。

### 目标用户

- 不想提前做攻略的旅行者
- 临时起意型用户
- 更关注“现在去哪”而不是“整趟行程怎么排”的用户

### 核心价值

- 降低临时决策成本
- 把“问 AI”“查地图”“看店铺”合并成一个流程
- 让推荐结果具备地理上下文，而不是纯文字建议

## 三、我解决的核心问题

### 1. 旅行类 AI 产品很容易停留在“会聊天”，但不真正可执行

如果 AI 只能说“你可以去东京塔附近逛逛”，那只是建议，不是产品能力。

所以这个项目里，我把聊天结果真正接到了地图和地点数据上：

- 能识别位置
- 能查附近地点
- 能输出具体店铺卡片
- 能在地图上展示 marker
- 能进入详情页看更多信息

### 2. “自然语言推荐”很容易和真实地理语义错位

比如用户说：

`我在东京塔边上，有没有商场可以逛逛`

如果直接把“东京塔”当搜索目标，系统就可能把东京塔本身、晴空塔之类景点也推荐出来。

这个项目专门处理了这类问题：

- 把“东京塔”识别成位置上下文，而不是推荐目标
- 把真正的搜索意图提纯为“商场 / 购物”
- 对候选结果做类型过滤和距离约束

### 3. 推荐结果不应该只是“这一轮”，而应该支持持续收窄

真实用户不会一开始就把需求说完整，更多是这样：

- `我在涩谷附近，有便利店吗`
- `有没有更安静一点的`
- `更近一点`

所以这个项目支持：

- 初次推荐
- 换一批
- 追加条件
- 历史批次保留
- 地图和不同批次结果的持续联动

## 四、核心功能

### 1. 基于当前位置的即时推荐

支持两种位置来源：

- 浏览器定位
- 用户输入的位置关键词或地标，例如“涩谷站”“东京塔”

系统会结合位置和自然语言需求，生成附近搜索。

### 2. 聊天式输入

用户不需要用固定筛选器，而是可以直接输入自然语言，例如：

- `我在涩谷站附近，想吃烧肉`
- `我刚到新宿，找个咖啡店坐一下`
- `我在东京塔附近，想顺路逛个展或美术馆`

### 3. 推荐卡片 + 地图联动

推荐结果不是纯文本，而是结构化卡片：

- 店名
- 距离
- 营业状态
- 评分
- 推荐理由

同时地图会显示对应 marker，支持 hover 联动。

### 4. 换一批

当用户不满意当前结果时，可以直接继续拿新的推荐批次，而不是重新发起完整搜索。

### 5. 追加条件

支持对上一轮搜索结果进行再收窄，例如：

- `更近一点`
- `预算更低`
- `更安静`
- `想坐露台`

### 6. 店铺详情页

详情页支持展示：

- 地址
- 营业时间
- 路线预览
- AI 评价速览
- 近期评论
- 打开 Google Maps

## 五、支持的推荐类型

这个项目并不只支持餐厅和咖啡店。

当前已经覆盖或可稳定处理的类型包括：

- 餐厅
- 咖啡店
- 酒吧
- 商场
- 便利店
- 酒店 / 住宿
- 景点
- 美术馆 / 博物馆
- 服装店 / 买手店 / 古着店
- 唱片店
- 偏“出片 / 拍照 / 打卡”类的地点探索需求

其中一部分通过显式的类型映射处理，另一部分通过自然语言搜索词转换和 Google Places Text Search 共同完成。

## 六、产品设计上的关键思考

### 1. 不是做“搜索框”，而是做“情境理解”

用户说的往往不是标准关键词，而是带上下文的话：

- `东京塔边上`
- `刚到新宿`
- `现在想坐一下`
- `顺路逛个展`

这类表达背后有三个维度：

- 人在哪
- 现在想做什么
- 这是不是上一轮推荐的延续

项目里最核心的一层其实不是 UI，而是这层意图拆解。

### 2. 地图不能只是展示结果，必须和对话强绑定

如果聊天和地图是两个松散模块，用户很快就会失去上下文。

所以这个项目里，我专门处理了：

- 不同推荐批次的 marker 保留
- 历史推荐消息 hover 时地图切回对应批次
- 单个卡片 hover 时地图高亮对应地点

这让“聊天流”和“空间流”真正连接起来。

### 3. 追加条件一定要继承上下文

用户说“更安静一点”，并不是发起一个全新搜索，而是在前一轮语义上继续筛选。

为此我做了：

- follow-up 识别
- 上下文 query 继承
- 类型继承，例如便利店、商场等不会在第二轮丢失
- 半径与过滤策略继续生效

## 七、技术实现

### 技术栈

- Next.js 16
- React 19
- TypeScript
- Tailwind CSS 4
- App Router
- Route Handlers

### 外部服务

- Google Maps JavaScript API
- Google Geocoding / Places / Route 相关接口
- 硅基流动兼容接口（LLM）

### 目录结构

```text
src/
  app/
    api/
  components/
  context/
  hooks/
  lib/
  types/
```

### 主要模块

#### 1. `/api/chat`

负责：

- 对话主流程
- 判断是否需要客户端搜索
- 判断是否是“换一批”
- 判断是否是“追加条件”
- 生成推荐理由相关的 LLM 输出

#### 2. `/api/geocode`

负责地理编码与反向地理编码。

#### 3. `/api/places`

负责附近地点搜索。

#### 4. `/api/place-details`

负责地点详情补全。

#### 5. 前端核心 hooks

- `useLocation`：定位、地理编码、服务区判断
- `useChat`：聊天状态、推荐结果、搜索流程、缓存、追加条件、换一批

## 八、工程上的关键处理

这个项目不仅是功能实现，我还专门处理了几个很典型的工程问题。

### 1. Google API Key 暴露风险

一开始前端直接请求 Google Places，key 会暴露在浏览器里。

后面我做了服务端代理，把以下请求尽量收回到服务端：

- geocode
- places search
- place details
- place photo
- route

同时保留了开发环境 fallback，避免本地环境因为 server-side 无法稳定访问 Google 而完全不可调试。

### 2. 生产构建稳定性

项目后期收尾时，真正挡上线的并不是功能，而是构建和类型问题：

- `next/font/google` 外网依赖导致 build 不稳定
- Google Maps typings 与运行时 API 不一致导致 TypeScript 报错

我做了两件事：

- 去掉外网字体依赖
- 对 Google Maps 相关类型做受控放宽，确保生产构建通过

### 3. 搜索结果跑偏

这是这个项目里很典型的业务工程问题。

例如：

- “东京塔”被当成推荐目标而不是位置上下文
- “涩谷便利店”被排到新宿区
- 第二轮“更安静一点”丢失第一轮的类型约束

我针对这些问题加了：

- 搜索词归一化
- 类型过滤
- 半径限制
- 距离优先排序
- append constraint 继承类型和半径

### 4. 历史结果与地图联动

推荐产品很容易只处理“当前这批结果”，但用户会不断产生新批次。

为了让历史批次仍可交互，我做了：

- 历史 places 合并保留
- 推荐批次级别的地图切换
- 避免 hover 时被最新一批错误覆盖

如果站在求职作品的角度，这个项目体现的不是单点编码能力，而是完整产品工程能力：

- 推荐理由生成
- 评论摘要生成

### 4. 工程落地能力

我处理了从 MVP 到可上线测试版过程中真实会遇到的问题：

- key 安全

```

```

## 十一、环境变量

在 `.env.local` 中配置：

```bash
SILICONFLOW_API_KEY=your_siliconflow_key
GOOGLE_MAPS_API_KEY=your_google_maps_key
NEXT_PUBLIC_GOOGLE_MAPS_API_KEY=your_google_maps_key
```

可选：

```bash
SILICONFLOW_BASE_URL=https://api.siliconflow.cn/v1
SILICONFLOW_CHAT_MODEL=deepseek-ai/DeepSeek-V3
SILICONFLOW_JSON_MODEL=deepseek-ai/DeepSeek-V3
UPSTASH_REDIS_REST_URL=your_upstash_redis_rest_url
UPSTASH_REDIS_REST_TOKEN=your_upstash_redis_rest_token
```

如果配置了 Redis，服务端会把以下缓存升级为“跨用户、跨实例共享”：

- Google Places 搜索结果
- 店铺详情
- JSON 型 LLM 任务结果（如推荐语、评论总结）

## 十二、部署

当前最适合的部署方式是 **Vercel**。

因为这是一个 Next.js App Router 项目，前端页面和 `src/app/api/*` 后端接口可以一起部署，不需要拆前后端。

部署时需要配置：

```bash
SILICONFLOW_API_KEY
GOOGLE_MAPS_API_KEY
NEXT_PUBLIC_GOOGLE_MAPS_API_KEY
```
